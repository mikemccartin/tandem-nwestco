## 1. Conversion Goals Hierarchy

### 1.1 Primary Conversion Goals (Ranked by Business Priority)

#### Goal 1: Emergency Service Requests (HIGHEST URGENCY)
**Business Value:** $500-$5,000+ per call, immediate revenue
**User Intent:** Equipment failure causing downtime, losing revenue NOW
**Response Requirement:** 10-minute callback, 2-4 hour technician arrival
**KPI:** Emergency form submissions + emergency hotline calls

**Why This Is Priority 1:**
- Highest margin service (premium pricing for emergency)
- Customer in pain = highest conversion rate
- Demonstrates core value proposition (maximize uptime)
- Builds loyalty through crisis response

**Conversion Path:**
1. User recognizes emergency (pump down, car wash broken, leak detected)
2. Google search: "emergency fuel pump repair near me"
3. Lands on site, sees red "EMERGENCY SERVICE" button immediately
4. One-click access to emergency form OR phone number

## 3. Form Design Philosophy

### 3.1 Core Principles

**1. Minimize Friction at Every Step**
- Fewest fields possible (question necessity of each field)
- Multi-step over single long form (reduce intimidation)
- Smart defaults and auto-fill (reduce typing)
- Clear labels and placeholders (reduce confusion)

**2. Progressive Disclosure**
- Capture email FIRST (enables abandonment recovery)
- Ask qualifying questions SECOND (route to right team)
- Collect contact details THIRD (only after commitment shown)
- Optional fields LAST (don't block submission)

**3. Build Trust Throughout**
- Privacy statement visible ("We'll never share your information")
- Response time commitment ("We'll respond within 4 hours")
- Security indicators (SSL badge, "Secure form" text)
- Progress indicator (show "Step 2 of 3")

**4. Mobile-First Design**
- Large touch targets (minimum 44x44px)
- Single-column layout (no side-by-side fields)
- Auto-focus on first field (reduce taps)
- Native input types (tel, email, date pickers)
- Sticky submit button (always accessible)

**5. Intelligent Validation**
- Inline validation (immediate feedback)
- Helpful error messages (not just "Invalid")
- Format examples in placeholder ("555-123-4567")
- Success indicators (green checkmark on valid input)

---

### 3.2 Form Field Best Practices

#### Required vs. Optional Fields Philosophy

**Required Fields (Minimum to Qualify Lead):**
- Email address (for follow-up and save/resume)
- Name (first name minimum, full name preferred)
- Phone number (for callback)
- Location (city/state or zip for routing)
- Service need description (what they need)

**Optional Fields (Nice-to-Have):**
- Company name
- Address (unless service requires site visit)
- Budget range (useful but may deter if too low)
- Timeline (helpful but can ask on call)
- How did you hear about us (marketing attribution)

**Never Required:**
- Home address (privacy concern)
- Detailed company information (too intrusive)
- Multiple phone numbers (one is enough)
- Referral source (intimidating)

---

#### Field Types & Validation Rules

**Email Address:**
- Input type: `email` (triggers email keyboard on mobile)
- Validation: RFC 5322 standard (allow + and subdomains)
- Error message: "Please enter a valid email address (e.g., you@company.com)"
- Auto-suggest common domains (@gmail.com, @outlook.com)
- Required: YES (captured in Step 1)

**Phone Number:**
- Input type: `tel` (triggers numeric keyboard on mobile)
- Format: (555) 123-4567 (auto-format as user types)
- Validation: 10 digits minimum (allow international +1)
- Error message: "Please enter a 10-digit phone number"
- Required: YES

**Name:**
- Input type: `text`
- Label: "Full Name" (one field) or "First Name" + "Last Name" (two fields)
- Validation: Minimum 2 characters, no numbers
- Error message: "Please enter your name"
- Required: YES (first name minimum)

**Location (City, State):**
- Input type: Two separate fields OR combined autocomplete
- City: Text input with autocomplete
- State: Dropdown with 6 Western US states (WA, OR, ID, MT, CA, AR)
- OR Zip Code: 5-digit input with auto-lookup for city/state
- Validation: Must match valid city/state combination
- Error message: "Please enter your city and state"
- Required: YES (for branch routing)

**Company Name:**
- Input type: `text`
- Validation: None (free-form)
- Required: NO (optional)

**Service Type/Market:**
- Input type: `select` dropdown
- Options:
  - Fuel Systems (Retail Fuel Station)
  - Fuel Systems (Commercial/Fleet)
  - Fuel Systems (Aviation)
  - Fuel Systems (Industrial)

  - Other
- Required: YES (for routing)

**Timeline:**
- Input type: `select` dropdown
- Options:
  - Emergency (need service now)

  - Within 1 month
  - Within 3 months
  - Planning stage (6+ months)
  - Not sure
- Required: YES (for prioritization)

**Description/Message:**
- Input type: `textarea`
- Placeholder: "Tell us about your project or service need..."
- Character limit: 1,000 characters (display counter)
- Validation: Minimum 20 characters
- Error message: "Please provide a brief description (at least 20 characters)"
- Required: YES

## 4. Multi-Step Form Strategy

### 4.1 Why Multi-Step Forms

**Benefits:**
- 30-50% higher completion rates vs. single long form
- Less intimidating (one question at a time)
- Better mobile experience (one screen per step)
- Enables progressive qualification
- Allows save-and-resume
- Higher quality leads (commitment increases with each step)

**Research Supporting Multi-Step:**
- Vendasta study: 300% increase in conversions for multi-step vs. single-step
- HubSpot data: Multi-step forms have 13.85% higher completion rate
- Mobile users: 25% higher completion on multi-step forms

## 5. Routing & Lead Assignment

### 5.1 Automatic Branch Assignment Logic

**Geographic Routing (Based on Location Field):**

Branch routing is automatic based on city/state input in quote/service request forms.

**Branch Service Areas:**

1. **Spokane, WA** (Eastern WA, Northern ID)
   - Washington: Spokane, Spokane Valley, Cheney, Liberty Lake
   - Idaho: Coeur d'Alene, Post Falls, Hayden, Moscow, Lewiston
   - Coverage radius: ~150 miles

2. **Portland, OR** (Western OR, SW WA)
   - Oregon: Portland, Salem, Eugene, Corvallis, Bend
   - Washington: Vancouver, Longview, Camas
   - Coverage radius: ~150 miles

3. **Boise, ID** (Southern ID, Eastern OR)
   - Idaho: Boise, Meridian, Nampa, Twin Falls, Idaho Falls
   - Oregon: Ontario, Baker City
   - Coverage radius: ~200 miles

4. **Kalispell, MT** (Headquarters - Northwest MT)
   - Montana: Kalispell, Whitefish, Columbia Falls, Libby
   - Coverage radius: ~100 miles

5. **Billings, MT** (Central/Eastern MT)
   - Montana: Billings, Missoula, Great Falls, Bozeman, Helena
   - Coverage radius: ~250 miles (largest territory)

6. **Fresno, CA** (Central CA)
   - California: Fresno, Visalia, Clovis, Madera, Hanford
   - Coverage radius: ~100 miles

7. **Bakersfield, CA** (Southern Central CA)
   - California: Bakersfield, Delano, Taft, Tehachapi
   - Coverage radius: ~75 miles

8. **San Jose, CA** (Bay Area, Northern CA)
   - California: San Jose, San Francisco, Oakland, Sacramento, Stockton
   - Coverage radius: ~150 miles

9. **Springdale, AR** (Northwest AR)
   - Arkansas: Springdale, Fayetteville, Bentonville, Rogers, Fort Smith
   - Coverage radius: ~100 miles

**Fallback Routing:**
- If city/state doesn't match exact coverage area: Calculate zip code distance to all branches
- Assign to nearest branch by driving distance (not straight-line)
- Display: "Your request will be handled by our [Nearest Branch] branch, approximately [X miles] from your location"

## 7. Mobile Conversion Optimization

### 7.1 Mobile User Behavior Insights

**Mobile Traffic Assumptions (Industry Data):**
- 60-70% of website traffic is mobile
- 40-50% of form submissions are mobile
- Mobile users have 25% lower conversion rates than desktop (if not optimized)
- Field personnel (technicians, operators) use phones primarily

**Mobile User Context:**
- On-the-go: At job site, in vehicle, during crisis
- Thumb-typing: One-handed input common
- Shorter attention span: Need to complete task quickly
- Network variability: May have slow connection
- Smaller screen: Less context visible at once

**Mobile Optimization Goal:**
- Match or exceed desktop conversion rates
- Reduce mobile form completion time to under 4 minutes
- Minimize scrolling and zoom requirements
- Enable one-handed operation where possible

---

### 7.2 Mobile-First Design Principles

**Layout:**
- Single-column forms (no side-by-side fields)
- Full-width input fields (edge-to-edge with padding)
- Large touch targets (minimum 44x44px, prefer 56x56px)
- Generous spacing between fields (16px minimum)
- Sticky submit button (always accessible, no scrolling to submit)

**Input Optimization:**
- Native keyboard types:
  - `type="email"` â†’ Email keyboard with @ symbol
  - `type="tel"` â†’ Numeric keypad
  - `type="number"` â†’ Number keyboard
  - `type="date"` â†’ Native date picker
  - `type="url"` â†’ URL keyboard with .com key
- Auto-focus first field (reduce taps)
- Auto-advance on completion (e.g., phone number â†’ next field)
- Format as user types (phone: (555) 123-4567)

**Validation:**
- Inline validation (immediate feedback, no page refresh)
- Error messages below field (not tooltip or modal)
- Success indicators (green checkmark when valid)
- Highlight field border (red for error, green for success)

**Form Submission:**
- Disable submit button during processing (prevent double-submit)
- Show loading spinner: "Submitting your request..."
- No page navigation (AJAX submit if possible)
- Success confirmation on same page (no redirect)

## 10. A/B Testing Roadmap

### 10.1 Test Prioritization Framework

**ICE Scoring (Impact, Confidence, Ease):**

Rate each test on scale of 1-10:
- **Impact**: How much will this improve conversion rate?
- **Confidence**: How confident are we this will work?
- **Ease**: How easy is it to implement?

**ICE Score = (Impact + Confidence + Ease) / 3**

**Priority Order:**
- Score 8-10: Test immediately (high priority)
- Score 6-7.9: Test within 1 month (medium priority)
- Score <6: Test eventually or skip (low priority)

---

### 10.5 A/B Testing Best Practices

**Test One Variable at a Time:**
- Don't change headline AND button color simultaneously
- Isolate variables to understand what caused change

**Statistical Significance:**
- Minimum 95% confidence level before declaring winner
- Use calculator: https://www.optimizely.com/sample-size-calculator/
- Don't end test early based on "looks like it's winning"

**Test Duration:**
- Run for at least 1 full week (accounts for day-of-week variance)
- Ideally 2-3 weeks for higher confidence
- If seasonal business, run for full business cycle

**Sample Size:**
- Minimum 100 conversions per variant (more for small lifts)
- For 5% baseline conversion rate, need ~2,000 visitors per variant
- Calculator: https://vwo.com/ab-split-test-duration/

**Document Everything:**
- Hypothesis (what you expect to happen)
- Variants (exact copy/design of each)
- Success metric (what you're measuring)
- Results (winning variant, lift %, significance)
- Learnings (why it worked or didn't)

**Implementation:**
- Use tool like Google Optimize, Optimizely, or VWO
- OR implement server-side A/B testing
- Ensure no "flicker" effect on page load
- Track variant in analytics for post-test analysis
